Flow Engine 集成手册

本手册面向后端与前端开发者，完整介绍如何将 Flow Engine（后端）与 flow-frontend（前端框架）集成到一个企业项目中。涵盖后端 Spring Boot 集成、前端 packages 集成能力、插件扩展机制，以及基于官方示例（后端 flow-engine-example、前端 app-pc / app-mobile）的集成分析。

适用版本：后端 0.0.52（Java 17 / Spring Boot 3.5.9）；前端 0.0.52（React 18 / TypeScript 5 / pnpm 10 monorepo）。

1. 整体架构与前后端关系

Flow Engine 采用前后端分离架构：

  • 后端（本仓库根目录）：Spring Boot 引擎，提供流程定义存储、流转、待办、查询等 REST API。
  • 前端（flow-frontend/ 子目录）：基于 React 的前端框架，提供流程设计器、PC/移动端审批组件、动态表单等可复用 packages，以及两个示例应用 app-pc / app-mobile。

2. 后端：引入依赖

2.1 一站式引入（推荐）

业务项目通常只需引入 flow-engine-starter，它会传递依赖 framework、infra、api、query 四个子模块：

<dependency>
    <groupId>com.codingapi.flow</groupId>
    <artifactId>flow-engine-starter</artifactId>
    <version>0.0.52</version>
</dependency>

starter 通过 Spring Boot 自动装配机制注册以下 AutoConfiguration：

自动配置类来源模块职责
com.codingapi.flow.AutoConfigurationflow-engine-starter注册 FlowService / WorkflowService / FlowRecordService / RepositoryHolderContext / 网关上下文 / 脚本上下文 / 延迟任务 Runner
com.codingapi.flow.infra.AutoConfigurationflow-engine-starter-infra注册 10 个 JPA 仓储实现，并通过 FlowJpaPackageRegistrar 注册 JPA 扫描包 com.codingapi.flow.infra
com.codingapi.flow.api.AutoConfigurationflow-engine-starter-api扫描 com.codingapi.flow.api，注册命令类 Controller（/api/cmd/**）
com.codingapi.flow.query.AutoConfigurationflow-engine-starter-query扫描 com.codingapi.flow.query，注册查询类 Controller（/api/query/**）

2.2 按需引入

若只想使用部分能力（例如只接查询、不暴露命令 API），可分别引入对应子模块：flow-engine-framework / flow-engine-starter-infra / flow-engine-starter-api / flow-engine-starter-query。

flow-engine-starter-infra 依赖 springboot-starter-data-fast，会自动引入 Spring Data JPA。若项目已自行引入 JPA，注意依赖版本一致性。

3. 后端：配置与数据库适配

Flow Engine 无强制配置项，需保证 Spring Data JPA 配置正确：

# 数据源（H2 示例）
spring.datasource.driver-class-name=org.h2.Driver
spring.datasource.url=jdbc:h2:file:./flow.db
spring.jpa.database-platform=org.hibernate.dialect.H2Dialect
spring.jpa.hibernate.ddl-auto=update
spring.jpa.show-sql=true

达梦 DM 数据库适配

项目已内置达梦方言支持，参考 flow-engine-example 的 application-dm.properties：

spring.datasource.url=jdbc:dm://localhost:5236
spring.datasource.username=SYSDBA
spring.datasource.password=SYSDBA001
spring.datasource.driverClassName=dm.jdbc.driver.DmDriver
spring.jpa.properties.hibernate.dialect=com.codingapi.example.dialect.HydDmDialect
# 需引入 DmJdbcDriver11 与 DmDialect-for-hibernate6.6

自定义方言 HydDmDialect 继承 Hibernate Dialect，针对达梦对 Hibernate 6.6 的部分语法做了修正。

4. 后端：必需实现的接口

引擎领域模型与具体用户体系解耦，集成方< strong>必须实现以下两个接口，否则引擎无法识别"谁在操作流程"以及"如何加载审批人"。

4.1 IFlowOperator（流程参与用户）

业务系统的用户实体应实现该接口：

public interface IFlowOperator extends IUser {
    long getUserId();
    String getName();
    boolean isFlowManager();                       // 是否流程管理员
    IFlowOperator forwardOperator(GroovyScriptRequest request); // 动态转交审批人
}

4.2 FlowOperatorGateway（操作者防腐层）

这是引擎访问用户体系的唯一入口，需注册为 Spring Bean：get(long id) 与 findByIds(List<Long> ids)。此 Bean 由 starter 的 GatewayContextRegister 注入到 GatewayContext 单例，引擎在加载审批人/转办人/加签人时统一通过它查询。实现该 Bean 是后端集成成功的关键。

4.3 当前操作者上下文

命令类 Controller 通过 UserContext.getInstance().current() 获取当前登录用户作为操作者。集成方需在鉴权过滤器中把认证后的 IFlowOperator 写入 UserContext（示例：flow-engine-example/.../security/MyAuthenticationTokenFilter.java）。调试时可在请求参数中传 operatorId=123 直接指定。

4.4 可选：脚本持久化 NodeViewJavaScriptRepository

节点中的路由/条件/表单权限逻辑由 Groovy 脚本驱动。脚本可内联在流程定义 JSON 中，也可独立持久化为命名脚本。若需后者，实现 NodeViewJavaScriptRepository（save/delete/get）并注册为 Bean，starter 的 NodeViewJavaScriptCacheContextRegister 会自动加载。

6. 前端：模块分层与 packages 全景

flow-frontend 是 pnpm monorepo，按"基础层 → PC/移动端组件 → 应用层"分层，跨 package 通过 workspace:* + 完整包名引用，禁止相对路径跨 package 引用。

外部依赖（非 workspace 包）

包版本作用

7. 前端：flow-core 基础能力集成

flow-core 是无 UI 的基础层，提供 5 个可被任意前端项目复用的能力：

能力核心 API集成用途
event-busEventBus.getInstance().on/emit/off(name, handler)跨模块松耦合通信（设计器通知审批刷新、节点变更、表单状态同步）
http-clientnew HttpClient(timeout, MessageBox, baseUrl?)，get/post/put/delete/page/download统一 API 出口，自动 token 鉴权与刷新、分页参数 Base64 编码、文件下载
presenterBasePresenter<S,M> + PresenterHooks.create()业务逻辑与视图分离，桥接 React useState 自动同步 state
view-bind-pluginViewBindPlugin.getInstance().register(name, Comp) / get(name)视图动态注册中心，设计器插件体系通过它注册节点视图/动作视图（运行时绑定而非硬编码）
groovy-formatterGroovyFormatter + GroovyScriptConvertorUtilGroovy 脚本格式化/压缩、自定义脚本标记与标题/元数据管理、return 表达式解析
集成方式：import { HttpClient, EventBus } from "@coding-flow/flow-core"。HttpClient 是审批组件对接后端的默认出口，业务项目可复用同一实例或自建。

8. 前端：流程设计器集成（flow-design）

flow-design 是基于 @flowgram.ai 固定布局编辑器构建的业务流程设计器，提供完整的可视化编排能力。

8.1 核心能力

能力说明
node-registry通过 FlowNodeRegistry 接口标准化定义并注册 19 种节点类型到 Flowgram 编辑器——13 主体节点（Start/End/Approval/Handle/Notify/Manual/Router/Trigger/Delay/SubProcess/Condition/Parallel/Inclusive）+ 6 分支 block 节点。每个节点带 meta（是否起始/结束、禁止删除/复制/添加/展开）
manager-group设计面板管理器组，把业务逻辑从 Presenter 抽离为独立类：WorkflowFormManager（表单字段 CRUD 与排序）、NodeConvertorManager（API↔渲染数据互转）、NodeManger（节点查询）、NodeRouterManager（路由节点展示/返回节点列表）、WorkflowStrategyManager（流程策略配置）

8.2 主要导出

flow-design 通过 src/index.ts 导出三大块：design-panel（设计面板，含节点列表、属性表单、版本管理、脚本编辑器、保存/导入/导出）、design-import（流程导入界面）、plugins（视图插件，如 Condition/Router 视图）。

8.3 集成方式

应用层引入 @coding-flow/flow-design，在页面中使用设计面板组件，并通过 API client 调用后端 /api/cmd/workflow/** 进行保存/加载/导入/导出。设计器内部通过 NodeConvertorManager 在 API JSON 格式与 Flowgram 渲染格式间互转。

9. 前端：审批组件集成（flow-pc / flow-mobile approval）

审批组件是业务系统对接流程流转的前端核心。PC 与移动端各有一套，均基于 flow-approval-presenter（Redux 状态管理框架）。

9.1 flow-pc-approval 导出

导出用途
ApprovalPanelDrawerPC 审批面板（抽屉形式），包含表单视图、流程节点历史、审批意见、动作按钮区
FlowTitle流程标题展示组件（带状态色）
WorkflowSelectModal流程定义选择弹窗（发起流程时挑选 workCode）
FlowMockMock 实例管理组件（创建/清理 mockKey，对接后端 /api/cmd/workflow/mock）
ActionFactory动作工厂单例，按 ActionType 渲染对应动作组件

9.2 ActionFactory 动作工厂

三端（PC / 移动 / 设计器表单）各有一份 ActionFactory 单例，按 8 种 ActionType 注册并渲染对应 React 组件：

// PC 端注册（packages/flow-pc/flow-pc-approval/.../action/factory.ts）
cache.set("PASS", PassAction);
cache.set("REJECT", RejectAction);
cache.set("RETURN", ReturnAction);
cache.set("TRANSFER", TransferAction);
cache.set("DELEGATE", DelegateAction);
cache.set("ADD_AUDIT", AddAuditAction);
cache.set("SAVE", SaveAction);
cache.set("CUSTOM", CustomAction);

// 使用：根据动作渲染按钮
const Comp = ActionFactory.getInstance().getFlowActionComponent(action);
扩展机制：要增加自定义审批按钮，通过 ActionFactory 注册 CUSTOM 类型组件，并在节点动作配置中启用。PC 与移动端 ActionFactory.render(action) / getFlowActionComponent(action) 各自返回对应平台组件。

9.3 审批面板内部结构

flow-pc-approval 内置组件：flow-approval-actions（动作按钮区）、flow-approval-content（表单内容）、flow-approval-sider（侧边流程图/节点）、flow-approval-title、flow-node-history（节点历史轨迹）、flow-time-node（时间轴）、form-view-component（表单渲染）。

移动端 flow-mobile-approval 导出 ApprovalPanel，提供精简的移动端审批界面。

9.4 API 对接

审批组件通过实现 FlowApprovalApi 接口（位于 flow-approval-presenter）对接后端，PC 端实现 FlowApprovalApiImpl 调用 /api/cmd/record/**：create / action / processNodes / revoke / urge。

10. 前端：表单引擎集成（form-engine）

业务流程表单数据渲染由外部 @coding-form/form-engine 提供，与 Flowgram 的 form-materials（节点属性配置）是两套独立表单能力：

表单能力来源用途
FormView@coding-form/form-engine渲染业务流程表单数据（string/integer/long/date/boolean 等）
createFormInstance同上编程式操作表单实例（取值/校验/赋值）
registerFormItems同上注册自定义字段类型组件——业务方扩展自定义表单项的入口>
app-pc/app-mobile 在启动时调用 registerFormItems() 注册自定义字段类型（见入口 register-form-types.tsx），并调用 registerImportFormView() 注册导入表单视图。

11. 前端：插件与扩展机制总览

核心原则：flow-frontend 大量采用"单例注册中心 + 运行时绑定"模式，使视图与逻辑解耦。集成方可通过注册而非修改源码来扩展能力。

16. 集成验证清单

后端

  • 项目正常启动，无 Bean 缺失异常

前端

  • pnpm install 成功，所有 workspace 包可构建（pnpm run build）
  • HttpClient baseURL 指向后端，跨域/CORS 已配置
  • 登录后可访问待办列表，数据来自 /api/query/record/todo

  • 流程设计器可创建/保存/加载流程定义

全栈联调

  • 在 PC 设计器编排一条请假流程并保存
