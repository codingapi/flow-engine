# Workflow 算法处理说明

相关的转换算法处理，就是对workflow对象数据的转换处理，存在两种workflow格式数据，分别是由@flowgram.ai的流程设计器产生的一种数据，和流程引擎flow-engine后台存储的数据格式。

这两种数据的最大差距就是在edge关系的处理上。

@flowgram.ai提供的数据是没有对edge关系进行描述信息，所有的edges关系都在nodes的数据过程中体现了，体现在nodes的先后顺序以及node下的blocks数据和先后顺序上。
而flow-engine存储的数据，则没有任何node下存在blocks的数据，所有的节点关系都通过edges的关系描述来表达。

为了实现这些过程的处理，一共提供了三个转换函数。
toNodes，即将带有blocks数据的@flowgram.ai格式workflow数据，转化为flow-engine后台服务的nodes数据。
toEdges，即将带有blocks数据的@flowgram.ai格式workflow数据，转化为flow-engine后台服务的edges数据。
toRender，即将后台flow-engine后台服务的edges数据和nodes数据转换为带有blocks数据格式的@flowgram.ai格式。

另外在处理转换的过程中还有一些特殊的情况说明：
一、 针对CONDITION, PARALLEL, INCLUSIVE这三种类型的节点，在flow-engine流程引擎后台是不存在的，也就是转换的过程中是不计算他们三个的节点和edge的，直接跳到他们的blocks的数据上计算相关的节点与边的关系。
在toRender的过程中，则需要将nodes中不存在的CONDITION, PARALLEL, INCLUSIVE这三种类型的节点给构建出来。

二、另外除了end节点不会存在下级节点以外，router节点也不存在下级节点。

三、@flowgram.ai表述节点的关系的时候，只有CONDITION, PARALLEL, INCLUSIVE这三种类型和其子节点（即CONDITION_BRANCH, PARALLEL_BRANCH, INCLUSIVE_BRANCH）下会存在blocks数据，其他节点的blocks数据都是空或者空数组。CONDITION, PARALLEL, INCLUSIVE这三种的blocks表述的节点关系都是并行的。但是CONDITION_BRANCH, PARALLEL_BRANCH, INCLUSIVE_BRANCH这三种节点下的blocks表述的则是顺序的，就是从如果blocks有两个，则会执行第一条再执行第二条。

四、当便利node的节点之所以能到了下一个node节点，是因为在最外循环下，则个node的子子blocks为空即视为是最外层循环的下一个node。
